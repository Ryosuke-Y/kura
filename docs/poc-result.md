## 検証日: 2026-03-15〜17

## 検証環境

- MacBook Air (M系チップ)
- Bun 1.3.6
- kuromoji.js 0.1.2 (IPA辞書)
- Budoux 0.8.0

## 詳細結果

### 1. bun:sqlite + FTS5 (PASS)

- FTS5仮想テーブルの作成・INSERT・MATCH全て正常動作
- `snippet()`, `bm25()` などのFTS5関数も使用可能
- AND/OR検索、ランキング検索が動作
- 特別な設定やビルドフラグ不要 — bun:sqliteにFTS5が標準搭載

### 2. kuromoji.js vs Budoux

| 項目 | kuromoji.js | Budoux |
|------|------------|--------|
| ロード時間 | 109-112ms | 1ms |
| 分割粒度 | 形態素（「知識」「管理」「ツール」） | フレーズ（「知識管理ツールは」） |
| 検索適性 | 高い（個別単語で検索可能） | 低い（フレーズ単位で検索不可） |
| 品詞情報 | あり（助詞除外などフィルタ可能） | なし |
| メモリ消費 | 約337MB（辞書展開後） | 約1MB |

**結論: 検索精度はkuromoji.js一択。ただしメモリが問題。**

Budouxは改行位置予測ツールであり、検索用途には粒度が粗すぎる。
kuromojiの辞書ロード時間（公称10-15秒）はBunでは112msに短縮され、問題にならなかった。

### 3. 日本語FTS5統合検索 (PASS)

分かち書きテキストをFTS5に登録し、検索クエリも分かち書きするアプローチが有効。

テスト結果（6/6 成功）:
- 「知識管理」→「知識管理ツールの比較」ヒット ✓
- 「日本語検索」→「SQLite全文検索の実装」ヒット ✓
- 「ローカルLLM」→「ローカルLLMの活用」ヒット ✓
- 「セキュリティ」→「エージェント時代のセキュリティ」ヒット ✓
- 「Markdown エディタ」（日英混在）→ ヒット ✓
- 「軽量 ナレッジ」（AND検索）→ ヒット ✓

設計上のポイント:
- 助詞・助動詞・記号を除外することで検索ノイズを軽減
- 登録時と検索時で同一のトークナイザーを使用（トークン一致を保証）
- `content_original`（表示用）と`content_tokenized`（検索用）を分離

### 4. パフォーマンス

**検索速度 (PASS — 基準の1000倍以上高速):**
- インメモリDB: 0.097ms/クエリ（1000件）
- ファイルDB: 0.47ms/クエリ（1000件）
- 基準100ms以下を大幅にクリア

**登録速度:**
- 0.14ms/件（1000件登録時）
- 1000件で約150ms

**メモリ (FAIL):**

```
起動直後:        34 MB
+kuromoji辞書: +337 MB = 371 MB  ← 主要因
+1000件FTS5:  +457 MB = 828 MB  ← トークナイズ一時オブジェクト含む
GC後:                    830 MB  ← GCで回収されない（辞書は保持必須）
```

メモリ問題の主因はkuromoji.jsのIPA辞書（メモリ展開後約337MB）。
FTS5インデックスのディスクサイズは0.76MB（1000件）と小さい。

---

## 判定: 設計変更付きGo

### 技術的には全て動作する

- bun:sqlite + FTS5: 完全動作、設定不要
- kuromoji.js + FTS5: 日本語検索が実用的精度で動作
- 検索速度: 基準の1000倍以上高速
- 辞書ロード: 基準の50倍以上高速

### メモリ戦略の見直しが必要

当初の「実行時50-80MB」目標はkuromoji辞書の実測値（337MB）と大幅に乖離。
ただし、CLIツールとしての利用パターンを考慮すると、影響範囲は限定的。

---

## メモリ問題の対策案

### CLI利用時（`kura search`, `kura index`）

プロセスは一時的に起動して終了するため、337MBの一時使用は実害が小さい。
32GB RAMマシンの1%程度であり、コマンド完了後にメモリは解放される。

### `kura serve`（Web UI常駐時）の対策

| 対策 | メモリ削減効果 | 実装難易度 | 検索品質への影響 |
|------|-------------|-----------|----------------|
| A. インデックス時のみkuromoji使用 | 大 | 中 | クエリ側のトークナイズ方法次第 |
| B. 子プロセスでトークナイズ | 大 | 中 | なし |
| C. 軽量辞書（UniDic Lite等） | 中 | 低 | 要検証 |
| D. トークナイズ結果キャッシュ | 中 | 低 | なし |
| E. Rustのlindera採用（将来） | 大 | 高 | 同等以上 |

### 推奨アプローチ

**Phase 1（本格開発）では対策Aを採用:**

1. `kura index` 時にkuromojiをロードし、全ノートを分かち書き→FTS5登録
2. `kura search` 時はkuromojiをロードし、クエリのみトークナイズ→MATCH
   - CLIなのでプロセスは一時的。337MBの一時使用は許容
3. `kura serve` 時は検索リクエストごとにクエリをトークナイズ
   - kuromoji常駐は避け、検索時のみ遅延ロードまたは子プロセス化を検討

**メモリ目標の修正案:**
- CLI実行時: 一時的に400MB以下（辞書ロード含む）→ 終了後0MB
- `kura serve` 常駐時: 50-80MB以下（辞書は非常駐）→ 検索時一時的に増加
